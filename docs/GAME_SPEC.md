# Claude Code 작업 지시서: Volfied 스타일 땅따먹기 게임 (React + TypeScript)

> 사용법: 이 파일을 프로젝트 루트에 `docs/GAME_SPEC.md`로 저장한 뒤
> Claude Code에서 `@docs/GAME_SPEC.md 를 읽고 마일스톤 M1부터 순서대로 구현해줘` 라고 지시하거나,
> 전체를 첫 프롬프트로 붙여넣어도 된다.

---

## 1. 역할과 목표

너는 20년차 시니어 게임/프론트엔드 엔지니어다. 1989년 타이토의 아케이드 게임
**Volfied(Qix 계열 땅따먹기)** 의 핵심 메커니즘을 충실히 재현한 웹 게임을
**React + TypeScript + Canvas 2D**로 구현한다.

- 게임 엔진 로직은 React와 완전히 분리된 **순수 TypeScript 모듈**로 작성한다 (헤드리스 테스트 가능).
- 외부 게임 엔진 라이브러리(Phaser, Pixi 등) 사용 금지. Canvas 2D API만 사용.
- 원작의 명칭/그래픽/사운드를 복제하지 않는다. 코드명은 **"Territory Raider"** 를 사용하고,
  모든 비주얼은 도형 기반 오리지널로 제작한다 (메커니즘만 차용).
- UI 텍스트는 **영문 전용**, 테마는 **다크(zinc/slate 팔레트 + 네온 액센트)**.

## 2. 게임 규칙 명세 (정확히 이대로 구현할 것)

### 2.1 플레이필드
- 논리 그리드: **가로 128 × 세로 96 셀** (상수로 분리, 변경 가능해야 함).
- 셀 상태: `BORDER`(최외곽 1셀 테두리, 시작부터 점령 상태) / `CLAIMED`(점령) / `UNCLAIMED`(미점령) / `TRAIL`(그리는 중인 선).
- 점령률 계산은 **내부 셀(BORDER 제외) 기준**: `claimedCells / interiorCells`.
- 캔버스 렌더링은 그리드를 정수배 스케일링 (예: 1셀 = 6px → 768×576).

### 2.2 플레이어 이동
- 4방향(상하좌우) 이동, 대각선 없음. 위치는 정수 셀 좌표.
- **실드 상태(기본)**: 플레이어는 "경계 셀" 위에서만 이동 가능.
  - 경계 셀 정의: `CLAIMED`(또는 `BORDER`) 셀 중 4방향 이웃에 `UNCLAIMED`가 하나 이상 있는 셀.
  - 실드 상태에서는 어떤 적과 접촉해도 죽지 않는다.
- **드로잉 상태**: 액션 키(Space)를 누른 채 `UNCLAIMED` 셀로 이동하면 진입.
  - 지나간 셀은 `TRAIL`로 마킹. 트레일은 **순서 보존 배열 + Set** 이중 자료구조로 관리.
  - 자기 트레일 셀로는 이동 불가 (자가 교차 금지). 직전 셀로 되돌아가기(후진)도 금지.
  - 드로잉 중 정지는 허용 (Qix와 달리 Volfied는 멈출 수 있음).
- 드로잉 중 다음 이동 셀이 `CLAIMED`/`BORDER`이면 트레일이 **폐합(commit)** 된다 → 2.3 실행.

### 2.3 영역 점령 알고리즘 (이 게임의 핵심 — 정확히 구현)
트레일 폐합 시:
```
1. 모든 TRAIL 셀 → CLAIMED 로 전환
2. 보스의 현재 셀을 시드로, UNCLAIMED 셀만 대상으로 4방향 flood fill
   → 도달 가능한 셀 집합 = reachable
3. UNCLAIMED 이면서 reachable 에 없는 모든 셀 → CLAIMED 로 전환
   (즉, "보스가 갇히지 않은 쪽"만 미점령으로 남고 나머지는 전부 점령)
4. 새로 CLAIMED 된 셀 위의 잡몹은 즉시 파괴 → 마리당 트랩 보너스 점수
5. 새로 CLAIMED 된 영역에 포함된 아이템 타일 → 자동 획득 (2.6)
6. 점령률 재계산. 한 번의 폐합으로 점령한 면적 비율에 따라 점수 부여 (2.7)
7. 진행 중이던 모든 스파크 제거
```
- 보스를 레이저로 처치한 상태(HP 0)라면 flood fill 시드가 없으므로 **전 영역 점령 = 즉시 스테이지 클리어 + 거대 보너스**.
- 트레일이 1셀 간격으로 평행하게 지나가며 생기는 슬리버(틈) 영역은 flood fill이 자연스럽게 점령 처리하므로 별도 예외 코드를 만들지 말 것.

### 2.4 적 (Enemies)
- **보스 (스테이지당 1기)**: `UNCLAIMED` 영역 안에서만 이동. 속도 벡터 + 경계(CLAIMED 셀) 충돌 시 반사 + 매 반사마다 ±15° 랜덤 섭동. `stageConfig.bossHp` 만큼 레이저 피격 가능.
- **잡몹 — Wanderer**: 미점령 영역 내부를 직선 이동 + 반사. 개수는 스테이지 설정값.
- **잡몹 — EdgeCrawler**: 점령 영역의 경계 셀을 따라 시계/반시계로 순회 (Qix의 Sparx 유사). 실드 상태 플레이어와는 충돌해도 무해, **드로잉 중인 플레이어와 접촉 시 사망**.
- **충돌 → 사망 조건 (드로잉 중에만)**:
  - 적(보스/잡몹)이 플레이어 본체와 접촉
  - 스파크가 플레이어 셀에 도달
- **스파크 메커니즘**: 보스 또는 잡몹이 `TRAIL` 셀과 접촉하면, 접촉 지점에 스파크 생성.
  스파크는 트레일 배열의 인덱스를 따라 **플레이어 방향으로** 일정 속도로 전진. 도달 시 사망.
  스파크 속도는 스테이지 설정값 (`sparkSpeed`).

### 2.5 사망 처리
- 라이프 1 감소. 트레일 전체를 `UNCLAIMED`로 롤백. 스파크 전부 제거.
- 플레이어는 **트레일 시작 지점(폐합 전 마지막 안전 셀)** 에서 리스폰, 2초 무적+점멸.
- 라이프 0 → Game Over 화면. 하이스코어는 `localStorage` 저장.

### 2.6 아이템
- 각 스테이지에 아이템 타일 3~5개를 미점령 영역에 가시적으로 배치 (스테이지 설정).
- 해당 타일이 점령 영역에 포함되는 순간 자동 획득. 효과:

| 코드 | 이름 | 효과 |
|---|---|---|
| T | Time Stop | 모든 적 10초 정지 |
| S | Speed Up | 플레이어 속도 ×1.5 (해당 스테이지 동안) |
| L | Laser | 레이저 5발 충전. 실드 상태에서 발사(미점령 방향), 잡몹 즉사 / 보스 HP -1 |
| P | Points | +5,000점 |
| C | Clear | 화면의 모든 잡몹 즉시 파괴 |

- 효과 수치는 전부 `constants.ts`에 상수로 분리할 것.

### 2.7 점수와 클리어
- 점령률 **80% 이상 → 스테이지 클리어**. 초과분 보너스: `(최종점령률 - 80) × 1,000`.
- 폐합 1회당 점수: `점령한 셀 수 × 10`, 단 한 번에 내부 면적의 **15% 이상** 점령 시 ×2 배율.
- 트랩 보너스(잡몹 가두기): 마리당 1,000점. 보스 레이저 처치 클리어: +50,000점.

### 2.8 스테이지 진행
- 기본 8스테이지. `stages.ts`에 설정 배열로 정의:
```ts
interface StageConfig {
  bossSpeed: number;      // cells/sec
  bossHp: number;
  wandererCount: number;
  edgeCrawlerCount: number;
  sparkSpeed: number;     // trail cells/sec
  itemTiles: ItemCode[];  // 이 스테이지에 등장할 아이템 종류
  bgColor: string;        // 점령 영역에 드러나는 배경색 (스테이지별 차별화)
}
```
- 스테이지가 올라갈수록 보스 속도/HP, 잡몹 수, 스파크 속도 증가.
- (보스전 확장) `bossFireCooldown?`(발리 간격), `projectileSpeed?`(탄속)도 스테이지별 상승.

### 2.8b 보스전 — "The Core fights back" (후속 확장)
- 보스는 쿨다운마다 **플레이어 조준 탄환**을 발사한다. 첫 발리는 스테이지 진입 후 한 쿨다운 뒤.
- **격노(rage)**: 점령률이 오를수록 보스가 거세진다 — 이기고 있다는 사실이 곧 위협.
  - rage 0 (<40%): 단발 조준탄, 기본 쿨다운
  - rage 1 (≥40%): 쿨다운 ×0.62
  - rage 2 (≥65%): 쿨다운 ×0.42 + **3-way 부채꼴**(±0.42rad)
- 탄환은 **미점령 공간에서만** 비행: 점령/경계 셀에 닿으면 소멸(스플래시) — *점령지가 곧 엄폐물*.
- 다른 적과 동일하게 **드로잉 중인 플레이어만** 피격(실드 무적 계약 유지), 리스폰 무적 존중.
- Time Stop(T)은 발사·비행 모두 동결(적 갱신과 같은 게이트).
- 사망 시 비행 중 탄환 전부 제거 + 최소 리스폰 무적 시간만큼 발사 유예.
- HUD: 보스 생존 중 CORE HP 핍 표시, rage 단계에 따라 색 상승(fuchsia→orange→rose).

### 2.9 조작
- 방향키: 이동 / Space(홀드): 드로잉 진입·유지 / X: 레이저 발사 / P: 일시정지 / Enter: 시작·재시작.

## 3. 기술 스택 및 아키텍처 제약

- **Vite + React 18 + TypeScript (strict: true)**. `any` 금지.
- 테스트: **Vitest(Jest 호환 API) + React Testing Library**. 엔진 로직은 DOM 없이 테스트.
- 스타일: Tailwind CSS. 다크 테마 고정 — zinc/slate 배경, 네온(cyan/fuchsia) 액센트.
- 상태 설계 (위반 시 재설계):
  - 게임 상태는 **React 외부의 엔진 인스턴스**가 소유. `useState`로 프레임 단위 상태를 들고 있지 말 것.
  - 엔진 → React 연결은 `subscribe`/`getSnapshot`을 노출해 **`useSyncExternalStore`** 로 구독.
    HUD(점수/라이프/점령률)는 값 변경 이벤트 시에만 갱신 (매 프레임 리렌더 금지).
- 게임 루프: `requestAnimationFrame` + **고정 타임스텝(60Hz) 누적기(accumulator)** 패턴.
  `setInterval` 사용 금지. 탭 비활성 복귀 시 거대 dt로 인한 터널링 방지 (dt 상한 클램프).
- 렌더링 2-레이어 전략 (필수):
  - **정적 레이어**: 그리드(점령/미점령/테두리)를 오프스크린 캔버스에 그려두고,
    **폐합 이벤트 발생 시에만** 다시 그린다. 매 프레임 메인 캔버스에 `drawImage` 1회.
  - **동적 레이어**: 플레이어/트레일/적/스파크/아이템/레이저만 매 프레임 드로우.
- 충돌 판정은 셀 좌표 기반 (O(1) 그리드 조회). 전체 엔티티 O(n²) 스캔 금지.

### 디렉토리 구조
```
src/
  engine/
    core/        # types.ts, grid.ts, gameState.ts
    systems/     # movement.ts, claim.ts, enemies.ts, spark.ts,
                 # items.ts, laser.ts, collision.ts, scoring.ts
    config/      # constants.ts, stages.ts
    index.ts     # createEngine(): { tick(input, dt), subscribe, getSnapshot, dispatch }
  ui/
    components/  # GameCanvas.tsx, Hud.tsx, TitleScreen.tsx,
                 # PauseOverlay.tsx, StageClearScreen.tsx, GameOverScreen.tsx
    hooks/       # useGameEngine.ts, useKeyboard.ts, useRafLoop.ts
  App.tsx / main.tsx
tests/           # 엔진 단위 테스트 중심
```
- SOLID 준수: 각 system 파일은 단일 책임. 엔진 코어는 렌더러를 모른다(DIP).
  적 타입 추가는 기존 코드 수정 없이 확장 가능하게(OCP) — 적 행동을 인터페이스로 추상화.

## 4. 마일스톤 (순서 엄수, 단계별 완결)

각 마일스톤 종료 시: `tsc --noEmit` 통과 + 테스트 전부 green + 실행 가능한 데모 상태 + 커밋.
실패 시 다음 단계로 넘어가지 말고 원인 분석 후 재구현.

- **M1 — 골격**: Vite 프로젝트 셋업, 그리드 모델, 2-레이어 캔버스 렌더링,
  경계 셀 위 플레이어 이동(실드 상태), 키보드 입력, 고정 타임스텝 루프.
- **M2 — 점령 코어**: 드로잉 상태, 트레일 규칙(자가교차/후진 금지), 폐합 감지,
  **flood fill 점령 알고리즘(2.3)**, 점령률 HUD. ← 이 단계의 단위 테스트가 가장 중요.
- **M3 — 적과 죽음**: 보스 반사 이동, Wanderer, EdgeCrawler, 충돌 판정,
  사망/트레일 롤백/리스폰/라이프, Game Over.
- **M4 — 스파크·아이템·레이저**: 스파크 추격, 아이템 5종, 레이저, 점수 시스템 완성.
- **M5 — 스테이지·화면**: 8스테이지 설정, 난이도 곡선, Title/Pause/Stage Clear/Game Over
  화면, 다크 테마 폴리시, 하이스코어 localStorage.
- **M6 — 마감**: 테스트 커버리지 보강(엔진 systems 위주), README(조작법/아키텍처 다이어그램),
  성능 점검(60fps 유지 확인, 정적 레이어 재드로우 횟수 로깅).

## 5. 필수 테스트 케이스 (M2~M4에서 작성)

1. **claim.test.ts**: 고정 픽스처 그리드에서 폐합 → 보스 반대편 영역만 점령되는지.
   보스가 큰 쪽/작은 쪽에 있는 두 시나리오 모두. 슬리버(1셀 틈) 자동 점령 검증.
2. 트레일 규칙: 자가 교차 시도 차단, 후진 차단, 폐합 시 TRAIL→CLAIMED 전환 검증.
3. 점령률 계산: BORDER 제외 내부 셀 기준인지.
4. 스파크: 트레일 접촉 지점에서 생성 → 인덱스 순서대로 플레이어 방향 전진 검증.
5. 사망 롤백: 트레일이 전부 UNCLAIMED로 복원되고 점령률이 변하지 않는지.
6. 보스 처치 시 전 영역 점령 + 클리어 트리거.
7. 아이템: 점령 영역에 포함되는 순간 1회만 획득되는지.

## 6. 금지 사항 (위반 시 해당 부분 재작성)

- React state로 매 프레임 게임 상태 갱신 / 엔진 로직을 컴포넌트 안에 작성
- 매 프레임 그리드 전체 셀 순회 드로잉 (정적 레이어 캐싱 없이)
- `setInterval` 기반 루프, `any` 타입, 매직 넘버 하드코딩 (상수 파일로)
- 원작 게임의 명칭·이미지·사운드 리소스 사용 (모든 에셋은 도형/코드 생성 오리지널)
- 한 마일스톤이 미완(테스트 실패) 상태에서 다음 마일스톤 착수
